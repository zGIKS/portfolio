When I took the Open Source course, I encountered something that was completely new for many of my classmates: most software documentation is written in Markdown. Most were accustomed to working on all their reports in tools like Word or similar, so the switch to a plain text format caused quite a bit of confusion at first.

The course required that the entire cycle report be done in Markdown, from document sections to tables, images, and longer structures. Additionally, everything had to be uploaded to GitHub, so it was also necessary to understand at least the basics of Git and versioning. For several classmates, this was complicated because they had never worked with repositories or version control tools.

Over time, I started to find a lot of sense in Markdown. It seemed cleaner and simpler to me than working with .docx files. I also liked the fact that it was portable and could be versioned without issues in Git. Compared to files generated by word processors, Markdown documents are much easier to maintain and review.

The problem arose when the document started to grow. The course syllabus was quite demanding, and maintaining large tables or complex structures in Markdown became tedious. A natural solution would have been to use LaTeX, but the professor didn't allow it at all. The other alternative was to use HTML within Markdown or rely on rendering tools.

Many classmates ended up using Pandoc or VS Code extensions to visualize their documents. Some of my friends really struggled to keep their tables and sections organized.

Looking for alternatives, I found some options. One of them was Obsidian, which is quite popular for working with Markdown. Personally, it never convinced me much. I don't know if it's my perception, but every time I hear about Obsidian, it reminds me of the typical Mac user working in a Starbucks showing off their coffee. Also, their monthly payment model didn't seem particularly attractive to me.

Finally, I found a tool that for me turned out to be much more comfortable: Typora. It's a fairly minimalist Markdown editor and very pleasant to write in. Although it costs around 20 dollars, the payment is one-time and the license is permanent, which seemed better to me than paying a monthly subscription.

![Markdown workflow image](https://res.cloudinary.com/dhwxayqeg/image/upload/v1772646586/ef22b01f-b3a4-4900-96ce-a07445598838.png)

One of the things I liked most was that the entire document style is controlled via CSS, so you can completely change the appearance. For example, you can make the document look like GitHub, use a dark theme, or adapt the format to academic styles. It also makes creating tables much easier, something that in pure Markdown can be quite uncomfortable.

While some classmates continued struggling with the manual syntax of their documents, I could generate much of the tables and structures quickly. Several times, some friends asked me for help with their documents because they were having difficulties keeping them organized.

For the final delivery, I used Pandoc to export the document. Something interesting is that Pandoc can respect the styles defined in CSS, so the final result looked practically the same as in the editor.

When I submitted the final work, the professor even thought the document wasn't made in Markdown because the format looked too clean. In the end, I showed him the tool and how I was working on the document.

The only part that still didn't convince me completely was the versioning workflow with Git. Typora doesn't have direct integration with Git, so I had to start working with terminal tools. I tried lazygit and tig, but none ended up fitting completely with the way I like to work.

For that reason, I started developing a small personal project called Nit, which aims to be a lighter and simpler alternative for working with Git from the terminal. It's still in development, but I already have a basic functional version. Probably in another post, I'll talk in more detail about that project.

If anyone knows light tools for working with Git or Markdown, I'd be quite interested in trying them. I'm always looking to improve my workflow.
